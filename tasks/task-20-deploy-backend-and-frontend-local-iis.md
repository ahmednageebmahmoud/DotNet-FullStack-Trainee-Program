## Goal

Publish the ASP.NET Core backend and Angular frontend, host both on local IIS, configure IIS application pools correctly, and make sure the browser can access the frontend and API without development servers.

---

## Required Technologies

| Technology | Description |
|---|---|
| **IIS** | Host the backend API and frontend locally |
| **ASP.NET Core Hosting Bundle** | Allow IIS to run ASP.NET Core applications |
| **Application Pool** | Isolate backend and frontend sites |
| **Angular Production Build** | Serve compiled frontend files from IIS |
| **Web.config** | Configure ASP.NET Core hosting and Angular route fallback |
| **IIS URL Rewrite** | Rewrite Angular client-side routes back to `index.html` |
| **Windows Permissions** | Allow IIS identities to read published files |

## IIS App Pool Tips

- Use separate application pools for the backend and frontend.
- For ASP.NET Core and Angular static hosting, set `.NET CLR Version` to `No Managed Code`.
- Use `ApplicationPoolIdentity` unless there is a clear reason to use a custom Windows account.
- Give the app pool identity read permission on the published backend and frontend folders.
- Restart the app pool after changing environment variables, permissions, or published files.
- If the backend writes logs or uploads files, give write permission only to the needed folder.
- Enable `Load User Profile` if the application needs user-profile based certificates, keys, or temp files.

---

## IIS Working Tips

- Enable IIS from Windows Features before starting the task.
- Install the ASP.NET Core Hosting Bundle after enabling IIS, then restart IIS.
- Install IIS URL Rewrite from `https://www.iis.net/downloads/microsoft/url-rewrite`.
- Make sure port bindings do not conflict with another IIS site or local development server.
- Add Windows Firewall allow rules if another device needs to access the local IIS site.
- Use `iisreset` only when needed; prefer recycling the specific app pool during normal work.
- Check IIS logs when requests fail before changing application code.
- Check Windows Event Viewer for ASP.NET Core startup errors.
- Confirm the published folder contains `web.config`.
- For Angular routes, install IIS URL Rewrite and configure fallback to `index.html` so browser refresh does not return `404`.
- In the frontend rewrite rule, skip real files and directories with conditions such as `IsFile` and `IsDirectory`.
- If the frontend and backend are hosted under the same IIS site, exclude `/api/*` from the Angular rewrite rule.
- For API CORS, allow the frontend IIS origin if backend and frontend use different ports or hostnames.

### Angular URL Rewrite `web.config` Hint

Use this as a starting point for the frontend `web.config`. Replace `<application name>` with the IIS application path when the Angular app is hosted under an application, or use `/index.html` when it is hosted at the site root.

```xml
<?xml version="1.0" encoding="UTF-8"?>
<configuration>
  <system.webServer>
    <rewrite>
      <rules>
        <rule name="Angular Routes" stopProcessing="true">
          <match url=".*" />
          <conditions logicalGrouping="MatchAll">
            <add input="{REQUEST_FILENAME}" matchType="IsFile" negate="true" />
            <add input="{REQUEST_FILENAME}" matchType="IsDirectory" negate="true" />
          </conditions>
          <action type="Rewrite" url="/<application name>/index.html" />
        </rule>
      </rules>
    </rewrite>
  </system.webServer>
</configuration>
```

## Validation Requirements

| Case | Expected Result |
|---|---|
| Backend site opens through IIS | API responds from the IIS URL |
| Frontend site opens through IIS | Angular app loads without `ng serve` |
| Browser refresh on Angular child route | IIS URL Rewrite returns `index.html` and the route still works |
| Static asset request | IIS serves the real file and does not rewrite it to `index.html` |
| API request | IIS sends the request to the backend and does not rewrite it to `index.html` |
| Login from frontend | Request reaches backend through IIS successfully |
| Protected API request | Auth token or cookie works through IIS |
| Invalid API request | Frontend still shows the correct error message |
| Backend startup failure | Error is visible in Event Viewer or IIS logs |
| Permission issue | Fixed by granting access to the app pool identity, not by using broad permissions |

## Tips

1. Deploy to clean publish folders instead of pointing IIS to source code folders.
2. Keep backend and frontend ports documented in the task submission.
3. Do not commit production secrets or local machine passwords.
4. Add screenshots or notes showing both IIS sites, app pools, and working URLs.
5. Add the frontend `web.config` rewrite rule to source control if the Angular build process does not generate it automatically.
